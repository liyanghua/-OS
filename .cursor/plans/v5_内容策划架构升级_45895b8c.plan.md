# V5 内容策划架构升级方案

## 当前系统诊断

### 已建成的核心资产

- **完整对象链**：OpportunityCard -> Brief -> TemplateMatch -> Strategy -> Plan -> Generation -> AssetBundle
- **编排引擎**：`OpportunityToPlanFlow` 支持会话缓存、局部重生成、SQLite 持久化、血缘追踪
- **Council 机制**：SSE 流式讨论 + 结构化 Proposal + Apply/Reject + asset_diffs
- **评估体系**：baseline/upgrade 对比 + LLM/规则双轨打分 + comparison 报告
- **记忆系统**：AgentMemory FTS5 + 多维检索 + council_memory_block
- **模板库**：6 个已编译模板 + index.json + TemplateMatcher
- **真实数据**：5 条 MediaCrawler XHS JSONL（桌布品类）+ 评论数据

### 四个战略断点

1. **生产断点**：机会卡 -> 发布内容的端到端路径有 5+ 人工断点；LLM 降级时产出质量差；无质量把关闭环
2. **能力断点**：`SkillRegistry` 元数据完整但 `execute_skill` 零调用；`ToolRegistry` 空注册；MCP 占位
3. **协作断点**：文字是 pre-wrap 纯文本；图片是描述文本无可视化；无真实的人+Agent 共编辑
4. **飞轮断点**：`WinningPattern`/`FailedPattern` 零写入；`TemplateEffectivenessRecord` 无引用；三条反馈管道不统一

## 问题 1：Production-Ready 内容生产流水线（最高优先级）

### 目标

从已有的 5 条真实 XHS 笔记出发，跑通「一键生产可直接发布的小红书笔记」，并在每一步生成可解释的质量评分。

### 当前最短路径分析

```
search_contents_2026-04-02.jsonl (5条真实笔记)
    |
    v
xhs_opportunity_pipeline.py (离线: parse -> 信号提取 -> 本体映射 -> 编译机会卡)
    |
    v  (缺失: opportunity_cards.json + pipeline_details.json 未在 data/ 下找到)
IntelHubAdapter -> XHSReviewStore.sync_cards_from_json
    |
    v  (断点: 需要人工 review 才能 promoted)
BriefCompiler.compile (只用 source_notes[0])
    |
    v
TemplateMatcher.match + RewriteStrategyGenerator.generate (LLM 优先)
    |
    v
NewNotePlanCompiler.compile
    |
    v  (三路并行, 各60s超时)
TitleGenerator / BodyGenerator / ImageBriefGenerator
    |
    v
AssetAssembler.assemble -> AssetBundle (策划交付物, 非发布物)
```

### 升级方案：Production Pipeline v5

#### Phase 1: 数据基底补全（让流水线能跑起来）

**目标**：从 5 条真实 JSONL 一键跑出 opportunity_cards.json + pipeline_details.json

- 在 [apps/intel_hub/workflow/xhs_opportunity_pipeline.py](apps/intel_hub/workflow/xhs_opportunity_pipeline.py) 的 `main()` 中确认 JSONL 路径可配指向 `data/fixtures/mediacrawler_output/xhs/jsonl/`
- 执行一次离线 pipeline，产出卡片到 `data/output/xhs_opportunities/`
- 验证 `IntelHubAdapter.get_source_notes` 能正确索引到 pipeline_details

**自动晋级快速通道**：

- 在 [apps/intel_hub/services/opportunity_promoter.py](apps/intel_hub/services/opportunity_promoter.py) 增加 `auto_promote_for_dev(opportunity_id)` 方法
- 当 `ENVIRONMENT=dev` 时允许跳过人工 review 直接 promote（保留阈值 override 参数）
- 这样 5 条笔记可以立刻进入策划流程

#### Phase 2: 一键编译 + 质量评分卡

**目标**：`compile_note_plan(opportunity_id, with_generation=True)` 一步到位，附带每阶段评分

当前 `build_note_plan` 已经串联 brief -> match -> strategy -> plan -> generation，但缺少：

- **质量评分卡**：在 `_run_generation` 之后，自动调用 `stage_evaluator` 对 brief/strategy/plan/asset 四阶段打分
- **质量门控**：如果某阶段分数低于阈值（如 brief < 0.6），自动触发 Council 讨论修正而非继续推进
- **编译报告**：返回结构化的 `CompilationReport`（每阶段评分 + 耗时 + 降级标记 + 质量风险）

新增 schema（在 [apps/content_planning/schemas/](apps/content_planning/schemas/)）：

```python
class CompilationReport(BaseModel):
    opportunity_id: str
    pipeline_run_id: str
    stage_scores: dict[str, StageEvaluation]  # brief/strategy/plan/asset
    quality_gate_passed: bool
    degraded_stages: list[str]  # LLM 降级的阶段
    total_time_seconds: float
    warnings: list[str]
    recommended_actions: list[str]  # "brief 缺少 evidence_summary, 建议补充"
```

在 [apps/content_planning/services/opportunity_to_plan_flow.py](apps/content_planning/services/opportunity_to_plan_flow.py) 的 `compile_note_plan` 末尾追加：

```python
# 编译完成后自动质量评估
report = self._evaluate_compilation(opportunity_id)
result["compilation_report"] = report.model_dump(mode="json")
```

#### Phase 3: 从 AssetBundle 到可发布格式

**目标**：AssetBundle 不再只是策划交付物，而是可以直接导出为小红书发布格式

当前 `AssetBundle` 包含：`title_candidates`, `body_draft`, `body_outline`, `image_execution_briefs`, `variant_sets`。

升级为 `PublishReadyPackage`：

```python
class PublishReadyPackage(BaseModel):
    asset_bundle_id: str
    platform: str = "xhs"
    
    # 最终选定的内容
    selected_title: str
    final_body: str  # 经过格式化的正文（含 emoji、分段、hashtag）
    cover_image_prompt: str  # 给图片生成工具的精确 prompt
    image_prompts: list[str]  # 内文图片 prompts
    
    # 小红书格式化
    hashtags: list[str]
    topic_tags: list[str]
    character_count: int
    
    # 质量报告
    compilation_report: CompilationReport
    
    # 可追溯性
    source_opportunity_id: str
    template_id: str
    strategy_id: str
```

在 [apps/content_planning/services/](apps/content_planning/services/) 新增 `publish_formatter.py`：

- `XHSPublishFormatter.format(asset_bundle, strategy) -> PublishReadyPackage`
- 自动选择最佳标题（按评分 + 规则）
- 正文格式化（段落/emoji/CTA/hashtag 自动插入）
- 从 strategy + plan 提取 hashtag/topic 建议

#### Phase 4: 差异化解释 -- "好在哪"

**目标**：让用户知道「这个生成结果好在哪，还能怎么优化」

核心思路：**对标源笔记做差异化对比**

- 新增 `QualityExplainer` 服务
- 输入：源笔记 + 生成结果 + 策略
- 输出：结构化的「改进说明」

```python
class QualityExplanation(BaseModel):
    vs_source_improvements: list[ImprovementItem]  # 相比源笔记的改进点
    strategy_alignment_score: float  # 策略执行一致性
    predicted_engagement_factors: list[str]  # 预测有效的互动因素
    optimization_suggestions: list[str]  # 可进一步优化的方向
    risk_flags: list[str]  # 潜在风险（如过度营销、语气不符等）
```

与 [apps/content_planning/evaluation/stage_evaluator.py](apps/content_planning/evaluation/stage_evaluator.py) 对接，复用 LLM 评委 + 规则维度。

## 问题 2：接入 GitHub 成熟 Skills/Tools

### 当前瓶颈诊断

```
SkillRegistry (元数据完整, 10条默认技能)
    |
    x  execute_skill() 零调用
    |
ToolRegistry (分发逻辑完整)
    |
    x  register_builtin_tools() 未被调用 -> 空注册
    |
MCPAdapter (结构占位, handler 返回 mcp_not_connected)
```

### 升级方案：三层工具接入架构

#### Layer 1: 激活内置工具注册（基础修复）

在 [apps/intel_hub/api/app.py](apps/intel_hub/api/app.py) 的 `create_app` 中：

```python
from apps.content_planning.agents.tool_registry import tool_registry, register_builtin_tools
register_builtin_tools()
```

对齐默认技能的 `executable_steps` 引用的工具名与实际注册名。

#### Layer 2: MCP 网关激活（接入外部工具）

将 MCP adapter 从占位升级为真实连接器。推荐接入的 GitHub 成熟工具方向：

- **内容增强类**：
  - SEO/关键词分析 MCP（如接入 5118/站长工具 API）
  - 竞品笔记分析工具
  - 热门话题/趋势追踪
- **图片生成类**：
  - Comfy UI / Stable Diffusion MCP server
  - 图片风格迁移工具
- **合规/质量类**：
  - 文本质量检测（敏感词、品牌词、语法）
  - 图片版权检测
- **发布类**：
  - 小红书发布 API 代理

架构设计：

```
LeadAgent / AssetProducerAgent
    |
    v
ToolRegistry.handle_tool_call(tool_name, arguments)
    |
    +-- builtin:compile_brief / generate_strategy / ...
    |
    +-- mcp:seo_keywords / trend_topics / image_generate / ...
    |
    +-- skill:full_pipeline / batch_variant / ...
```

在 [apps/content_planning/agents/mcp_adapter.py](apps/content_planning/agents/mcp_adapter.py) 中：

- `_discover_from_server` 接入真实 MCP SDK 的 `Client.list_tools()`
- handler 通过 MCP Client 转发 `tool_call`
- 配置文件 `config/mcp_servers.yaml` 定义可用 MCP 服务列表

#### Layer 3: Skill 可执行化（编排复合工具）

让 `execute_skill` 真正可用：

- 在 `load_defaults()` 中，确保每条技能的 `executable_steps` 只引用**已注册**的工具
- 为 `full_pipeline` 技能填充完整的 step chain：
`compile_brief -> match_templates -> generate_strategy -> compile_plan -> generate_titles -> generate_body -> generate_image_briefs -> assemble_asset_bundle`
- 在 LeadAgent 的 tool routing 中，当 LLM 选中一个 skill，调用 `skill_registry.aexecute_skill`

## 问题 3：混合工作台形态设计

### 设计原则

```
左侧：结构化流程面板（对象链 + 阶段导航 + 质量仪表盘）
右侧：自由画布预览（内容拼版 + 图文排版预览 + Agent 协作区）
底部：协同时间线 + 操作历史
```

### 架构分层

#### 左侧 -- 结构化流程面板（Structure Panel）

保持当前的阶段 Tab 结构，但升级为更紧凑的流程导航：

```
[Brief] -> [Strategy] -> [Plan] -> [Generation] -> [Asset] -> [Publish]
  0.82      0.75          0.88      生成中...       -          -
```

每个阶段显示：质量评分徽章 + 完成状态 + 阻塞标记

当前阶段展开详情卡片（Brief 字段、Strategy 维度、Plan 大纲等），支持**行内编辑** -- 这是 "人" 直接干预对象的入口。

#### 右侧 -- 自由画布预览（Canvas Preview）

**不是无限画布**（那太重了），而是**内容预览画布**：

- **手机模拟器框架**：预览生成的笔记在小红书的真实展示效果
  - 封面图预览（来自 image_execution_brief 描述或生成的图片）
  - 标题预览（带字符计数、关键词高亮）
  - 正文预览（带段落、emoji、hashtag 渲染）
  - 标签/话题预览
- **变体对比**：左右滑动对比不同标题/正文/图片方案
- **Agent 标注层**：Council 建议的改进点直接标注在预览上（类似 Figma 评论针脚）

技术实现建议：

- 基于当前 Jinja2 模板体系，新增一个 `_preview_canvas.html` include
- 手机框架用 CSS 模拟，不需要引入 Canvas/WebGL
- 变体对比用 CSS flexbox + 切换动画

#### 协同交互模式

```
人的操作:
  - 在左侧结构面板直接编辑字段（标题、正文、约束）
  - 在右侧预览上标注修改意见
  - 发起 Council 讨论

Agent 的响应:
  - 字段变更后自动触发下游 stale 标记 + 建议重新生成
  - 标注转化为 Council 问题自动发起讨论
  - 实时在预览上高亮受影响区域
```

#### 文字/图片/视频的差异化处理

- **文字**：行内编辑 + Markdown 渲染 + Agent 建议 diff
- **图片**：
  - 阶段1: ImageExecutionBrief 文本描述 + 风格参考图引用
  - 阶段2: 接入图片生成 MCP -> 真实图片预览
  - 支持「图生图」：上传参考图 -> Agent 分析风格 -> 生成执行 brief
- **视频**：
  - 当前不在 MVP 范围
  - 设计预留：视频脚本 = 分镜序列（图片 brief 列表 + 时间轴 + 旁白/BGM 指令）

## 问题 4：数据飞轮与资产沉淀闭环

### 当前断裂的三条反馈管道

```
管道A: POST /asset-bundle/{id}/feedback -> FeedbackRecord (plan_store)
管道B: POST /{id}/publish-result -> B2BPlatformStore + FeedbackRecord
管道C: POST /{id}/review-feedback -> ReviewLoop -> AgentMemory

WinningPattern / FailedPattern -> 零写入
TemplateEffectivenessRecord -> 零引用
```

### 飞轮架构设计

```
                    +-------------------+
                    |   发布到小红书     |
                    +--------+----------+
                             |
                    录入发布结果 (笔记ID + 互动数据)
                             |
                    +--------v----------+
                    | UnifiedFeedback   |
                    | (单一事实源)       |
                    +--------+----------+
                             |
            +----------------+----------------+
            |                |                |
   +--------v------+  +-----v-------+  +-----v--------+
   | Pattern       |  | Template    |  | Agent        |
   | Extractor     |  | Ranker      |  | Memory       |
   +--------+------+  +------+------+  +------+-------+
            |                |                |
   写入 Winning/   更新模板效果   写入 lesson/
   Failed Pattern  权重/排序     brand_preference
            |                |                |
            +--------+-------+--------+-------+
                     |                |
              +------v------+  +------v------+
              | 下一次策划时  |  | Council 讨论 |
              | 自动注入偏好  |  | 引用历史数据  |
              +-------------+  +-------------+
```

#### Phase 1: 统一反馈事实表

合并三条管道为 `UnifiedFeedbackStore`：

```python
class UnifiedFeedback(BaseModel):
    feedback_id: str
    opportunity_id: str
    asset_bundle_id: str
    template_id: str  # 关键: 关联回模板
    strategy_id: str  # 关键: 关联回策略
    
    # 发布数据
    platform: str = "xhs"
    published_note_id: str = ""
    published_at: datetime | None = None
    
    # 互动指标
    like_count: int = 0
    collect_count: int = 0
    comment_count: int = 0
    share_count: int = 0
    view_count: int = 0
    
    # 分级
    performance_tier: str = "unknown"  # excellent/good/average/poor
    engagement_score: float = 0.0  # 归一化 0-1
    
    # 人工备注
    human_notes: str = ""
    human_edits_summary: str = ""  # 人工改了什么
```

单一写入入口，触发三路下游：

1. `PatternExtractor.evaluate(feedback)` -> 条件写入 `WinningPattern` 或 `FailedPattern`
2. `TemplateRanker.update_effectiveness(feedback)` -> 更新模板排序权重
3. `AgentMemory.store(...)` + `ReviewLoop.process_feedback(...)` -> 写入记忆

#### Phase 2: Pattern 自动提取

激活 [apps/content_planning/schemas/feedback.py](apps/content_planning/schemas/feedback.py) 中已定义的 `WinningPattern` / `FailedPattern`：

```python
class PatternExtractor:
    def evaluate(self, feedback: UnifiedFeedback) -> None:
        if feedback.performance_tier == "excellent":
            # 从关联的 strategy + plan 提取成功模式
            pattern = self._extract_winning(feedback)
            self.store.save_winning_pattern(pattern)
        elif feedback.performance_tier == "poor":
            pattern = self._extract_failed(feedback)
            self.store.save_failed_pattern(pattern)
```

#### Phase 3: 模板效果权重闭环

激活 `TemplateEffectivenessRecord`：

- 每次发布结果录入时，创建 `TemplateEffectivenessRecord` 关联 `template_id`
- `TemplateRetriever` 在 `list_templates()` 时按效果权重排序
- 在 [data/template_extraction/templates/index.json](data/template_extraction/templates/index.json) 中增加 `effectiveness_score` 字段

#### Phase 4: 记忆增强策划

在策划流程各阶段注入历史教训：

- `BriefCompiler.compile()` 时，查询同品类/同品牌的 `WinningPattern`，注入 `suggested_direction`
- `RewriteStrategyGenerator.generate()` 时，查询 `FailedPattern`，注入 `avoid_directions`
- `TitleGenerator._build_user_prompt()` 时，从 AgentMemory 取历史高分标题特征
- Council 讨论时，`council_memory_block` 已支持，确保 `WinningPattern`/`FailedPattern` 也能被检索

### "越用越好"的量化验证

- 引入 `LearningDashboard`：展示模板效果权重变化曲线、策略评分趋势、Pattern 积累数量
- 每 N 次策划后自动生成 `LearningReport`：对比第 1 次 vs 第 N 次的管线平均质量分

## 技术约束与决策

- **不引入新的前端框架**：继续基于 Jinja2 + 原生 JS + SSE，保持当前架构的一致性
- **LLM 依赖管理**：所有 LLM 调用保留规则降级，但在 CompilationReport 中明确标记降级阶段
- **数据安全**：真实 XHS 数据只在本地使用，不上传到外部服务（MCP 工具只传结构化参数）
- **渐进式升级**：每个 Wave 独立可交付，不阻塞后续 Wave
