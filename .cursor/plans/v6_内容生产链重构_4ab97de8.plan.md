# V6 小红书内容生产链重构

## 设计原则

- **不新建与现有模型重叠的对象**：复用 `XHSNoteRaw`/`XHSParsedNote`、`XHSOpportunityCard`、`OpportunityBrief`
- **唯一真新增**：`ExpertScorecard`（8 维专家评分）+ `NoteToCardFlow`（上游编排）
- **评估器复用**：基于现有 `BaseStageEvaluator` 框架新增 2 个评估器，扩展 2 个现有评估器
- **API 延续现有前缀**：所有新端点挂在 `/content-planning/v6/`

## Phase 1: 模型层 — 扩展现有对象 + 新增 ExpertScorecard

### 1.1 扩展 XHSOpportunityCard（语义字段组）

在 [apps/intel_hub/schemas/opportunity.py](apps/intel_hub/schemas/opportunity.py) 的 `XHSOpportunityCard` 末尾新增 V6 字段组：

```python
# V6 语义升级字段
audience: str = ""                    # 目标人群画像
scene: str = ""                       # 使用场景
pain_point: str = ""                  # 用户痛点
desire: str = ""                      # 用户渴望
hook: str = ""                        # 内容钩子
selling_points: list[str] = Field(default_factory=list)
content_angle: str = ""               # 内容切入角度
format_suggestion: str = ""           # 格式建议
why_now: str = ""                     # 时效性理由
why_worth_doing: str = ""             # 值得做的理由
supporting_signal_ids: list[str] = Field(default_factory=list)
benchmark_refs: list[str] = Field(default_factory=list)
card_status: str = "draft"            # draft | ready_for_score | blocked
```

这些字段在 `opportunity_compiler.py` 编译时从 signal/scene/selling 数据中提取填充。

### 1.2 扩展 OpportunityBrief（production-ready 字段组）

### 1.3 新增 ExpertScorecard schema

新建 [apps/content_planning/schemas/expert_scorecard.py](apps/content_planning/schemas/expert_scorecard.py)：

- 8 个评分维度（每维度 0-1）：`demand_strength`, `differentiation`, `commerce_fit`, `scalability`, `reusability`, `visual_potential`, `video_potential`, `risk_score`
- 每维度附带 `evidence_sources: list[str]` + `explanation: str`
- `total_score`: 加权汇总（权重从 `config/scorecard_weights.yaml` 读取）
- `confidence`: 纯数据完整度指标（不看内容好坏）
- `recommendation`: `ignore | observe | evaluate | initiate`
- `score_reasons`, `blocking_risks`, `upgrade_advice`: list[str]
- 关联字段：`scorecard_id`, `card_id`, `opportunity_id`, `project_id`

### 1.4 新增 scorecard 权重配置

新建 [config/scorecard_weights.yaml](config/scorecard_weights.yaml)：

```yaml
dimensions:
  demand_strength: { weight: 0.18, label: "需求强度" }
  differentiation: { weight: 0.14, label: "差异化" }
  commerce_fit: { weight: 0.18, label: "商品适配" }
  scalability: { weight: 0.12, label: "可规模化" }
  reusability: { weight: 0.10, label: "可复用性" }
  visual_potential: { weight: 0.10, label: "视觉潜力" }
  video_potential: { weight: 0.08, label: "视频潜力" }
  risk_score: { weight: 0.10, label: "风险", inverse: true }

recommendation_thresholds:
  initiate: 0.7
  evaluate: 0.5
  observe: 0.3
  # below 0.3 -> ignore

confidence_components:
  field_completeness: 0.30
  comment_sample_size: 0.25
  benchmark_count: 0.25
  commerce_data: 0.20
```

## Phase 2: 评估器层 — 新增 2 个 + 扩展 2 个

### 2.1 新增 IngestEvaluator

在 [apps/content_planning/evaluation/stage_evaluator.py](apps/content_planning/evaluation/stage_evaluator.py) 新增：

- stage = `"ingest"`
- 4 维度：`field_completeness`（原始字段完整度）、`comment_coverage`（评论信号覆盖度）、`commerce_data`（商品承接信息）、`benchmark_coverage`（同类样本覆盖度）
- 规则逻辑：检查 `XHSParsedNote` + `pipeline_details` 的字段非空率
- 输出：`StageEvaluation`，并附带 `optimization_suggestions`（如"缺 OCR"、"缺商品映射"）

### 2.2 新增 ScorecardEvaluator

新增 stage = `"scorecard"` 评估器：

- 4 维度：`evidence_backing`（每维度是否有证据支撑）、`score_consistency`（总分是否被单一弱维度误导）、`recommendation_risk_alignment`（推荐与风险一致性）、`confidence_data_match`（confidence 与数据完整度匹配）
- 规则逻辑：交叉校验 scorecard 各字段的内部一致性

### 2.3 扩展 CardEvaluator

在现有 4 维度基础上新增 3 维度（总 7 维度）：

- `audience_clarity`（人群是否清晰，检查 `card.audience` 非空且 > 10 字符）
- `scene_specificity`（场景是否具体，检查 `card.scene`）
- `hook_strength`（钩子是否成立，检查 `card.hook`）

### 2.4 扩展 BriefEvaluator

新增 3 维度：

- `claim_evidence_alignment`（claim-evidence 对齐度）
- `visual_readiness`（视觉准备度，检查 `visual_direction` + `cover_direction` + `image_plan`）
- `production_completeness`（生产完整度，检查 V6 字段填充率）

## Phase 3: 服务层 — ExpertScorer + NoteToCardFlow

### 3.1 新增 ExpertScorer 服务

新建 [apps/content_planning/services/expert_scorer.py](apps/content_planning/services/expert_scorer.py)：

- `ExpertScorer.score(card: XHSOpportunityCard, note_context: dict) -> ExpertScorecard`
- 从 `config/scorecard_weights.yaml` 加载权重
- 每个维度的计算逻辑：
  - `demand_strength`: 收藏率 35% + 评论意图密度 35% + 深度互动率 20% + 时效性 10%
  - `differentiation`: 复用 `_build_strength_score` 中的跨模态分 + 标题新颖度启发式
  - `commerce_fit`: 商品关联 + 评论购买意图关键词
  - 其余维度用启发式规则，LLM 可用时升级
- `confidence` 计算：4 个数据完整度指标的加权平均
- `recommendation` 基于 `total_score` 阈值映射

### 3.2 新增 NoteToCardFlow 上游编排

新建 [apps/content_planning/services/note_to_card_flow.py](apps/content_planning/services/note_to_card_flow.py)：

编排链路：

```
run_ingest_eval(parsed_note)
  -> if blocked: return {blocked, suggestions}
compile_enriched_card(parsed_note, pipeline_result)
  -> run_card_eval(card)
  -> if blocked: return {blocked, suggestions}
score_card(card, note_context)
  -> run_scorecard_eval(scorecard)
  -> if recommendation in {evaluate, initiate}: auto_promote
  -> return {card, scorecard, evaluations}
```

门控规则（硬阻塞）：

- `IngestEval.status == block` -> 不生成 card
- `CardEval.status == block` -> 不生成 scorecard
- `Scorecard.recommendation == ignore` -> 不 promote

下游对接：promoted card 交给现有 `OpportunityToPlanFlow.build_brief()`

### 3.3 扩展 BriefCompiler

在 [apps/content_planning/services/brief_compiler.py](apps/content_planning/services/brief_compiler.py) 中：

- `compile()` 方法增加 `scorecard: ExpertScorecard | None = None` 参数
- 当 scorecard 存在时，从 scorecard 的 `upgrade_advice` 填充 `suggested_direction`
- 从 scorecard 维度分数驱动 V6 字段：高 `visual_potential` -> 填充 `visual_direction`；高 `commerce_fit` -> 强化 `cta`
- 设置 `brief.scorecard_id`

## Phase 5: API 层

在 [apps/content_planning/api/routes.py](apps/content_planning/api/routes.py) 新增 V6 路由组：

- `POST /v6/ingest-eval/{opportunity_id}` — 对原始笔记运行数据完整度评估
- `POST /v6/enrich-card/{opportunity_id}` — 用 V6 语义字段增强现有 card
- `POST /v6/score/{opportunity_id}` — 生成 ExpertScorecard
- `GET /v6/scorecard/{opportunity_id}` — 获取 scorecard
- `POST /v6/compile-brief/{opportunity_id}` — 基于 scorecard 编译 production-ready brief
- `POST /v6/run-pipeline/{opportunity_id}` — 一键全链路（ingest eval -> enrich -> score -> brief）
- `GET /v6/pipeline-status/{opportunity_id}` — 链路状态与各阶段 eval 结果
