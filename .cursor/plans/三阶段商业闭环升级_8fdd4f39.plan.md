# 小红书内容策划编译平台 — 三阶段商业闭环升级计划

## 现状摘要

### 已有骨架（可复用）

- 主链完整：`Opportunity -> Brief -> Strategy -> Plan -> AssetBundle`（[opportunity_to_plan_flow.py](apps/content_planning/services/opportunity_to_plan_flow.py)）
- 评估/对比体系：`StageEvaluation`、`PipelineEvaluation`、`ComparisonReport`（[evaluation.py](apps/content_planning/schemas/evaluation.py)、[comparison.py](apps/content_planning/evaluation/comparison.py)）
- B2B 多租户：`Workspace`/`BrandProfile`/`Campaign`/`WorkspaceMembership`/`ApprovalRecord`/`PublishResult`（[b2b_platform/schemas.py](apps/b2b_platform/schemas.py)、[b2b_platform/storage.py](apps/b2b_platform/storage.py)）
- 反馈雏形：`PublishedAssetResult`（[feedback.py](apps/content_planning/schemas/feedback.py)）+ `POST .../feedback` 路由（[routes.py L644-674](apps/content_planning/api/routes.py)）
- 版本追踪：`PlanLineage`（[lineage.py](apps/content_planning/schemas/lineage.py)）+ `append_version` / `load_versions`（[plan_store.py](apps/content_planning/storage/plan_store.py)）

### 关键缺口

- `PublishResult` 无 HTTP 入口（仅 `storage.record_publish_result`）；`feedback` 路由不写库
- 无 `FeedbackRecord` / `WinningPattern` / `FailedPattern` 模型
- `comparison` 只比中间评分，无 outcome 维度
- `BrandProfile` 只有轻量字段，无 `BrandGuardrail` / `BrandVoice` / `AudienceProfile`
- proposal/apply 无品牌阻断语义（`guardrail_warnings` / `blocked_by_guardrail`）
- 无 `ObjectAssignment` / `ObjectComment` / `WorkspaceTimelineEvent` / `ApprovalRequest`
- 对象状态机不统一，缺少 `in_planning` / `ready` / `approved` / `exported` / `published` 全链路状态
- Asset Workspace 缺发布结果录入与 outcome summary

## Phase 1：发布结果闭环（~4 周）

### 1.1 数据模型增补

在 [apps/content_planning/schemas/](apps/content_planning/schemas/) 下：

- **增强 `PublishResult`**（[b2b_platform/schemas.py L117](apps/b2b_platform/schemas.py)）：新增 `opportunity_id`、`brief_version`、`strategy_version`、`plan_version`、`asset_bundle_version` 字段
- **新建 `feedback_models.py`**：定义 `FeedbackRecord`（含 `approval_rounds`、`manual_edits_count`、`time_to_ready`、`published_at`、`engagement_proxy`、`feedback_quality`）、`WinningPattern`、`FailedPattern`

### 1.2 存储层

- [plan_store.py](apps/content_planning/storage/plan_store.py)：新增 `feedback_records`、`winning_patterns`、`failed_patterns` 三张 SQLite 表 + CRUD 方法
- [b2b_platform/storage.py](apps/b2b_platform/storage.py)：增强 `publish_results` 表 DDL 以容纳新版本绑定字段

### 1.3 对比体系升级

- [comparison.py](apps/content_planning/evaluation/comparison.py)：`ComparisonReport` 与 `StageDelta` 增加 `outcome` 维度（`approval_rounds_delta`、`edit_count_delta`、`time_delta`、`engagement_delta`），支持 baseline / single-agent / council 三路对比

### 1.4 API 路由

在 [routes.py](apps/content_planning/api/routes.py) 新增或增强：

- `POST /content-planning/asset-bundle/{id}/publish-result` — 写入 `PublishResult`（接通 `platform_store.record_publish_result`）
- `GET /content-planning/opportunities/{id}/outcome-summary` — 聚合该机会下所有 publish results + feedback
- `GET /content-planning/comparison/{id}/outcome-delta` — 含 outcome 的双层对比
- 修复现有 `POST .../feedback`：增加 SQLite 持久化

在 [app.py](apps/intel_hub/api/app.py) B2B 区：

- `GET /b2b/workspaces/{id}/feedback` — workspace 级反馈列表
- `GET /b2b/workspaces/{id}/pipeline` — 机会到发布漏斗数据

### 1.5 前端落点

- **增强 [content_assets.html](apps/intel_hub/api/templates/content_assets.html)**：
  - 「标记已发布」改为正式的「发布结果录入」表单（含 platform、external_ref、engagement 指标）
  - 新增 outcome summary 卡片（追溯到 Brief/Strategy/Plan 版本）
- **新增 `content_feedback.html` 模板**：展示 `PublishResult` / `FeedbackRecord` / `WinningPattern` 列表，机会到发布漏斗、各阶段返工/审批轮次、模板/策略/资产表现对比三视图

### 1.6 Phase 1 门槛验收

- 每个已发布资产都能追溯到对应 Brief/Strategy/Plan/AssetBundle 版本
- comparison 能同时回答"中间评分 + 发布结果"哪套更有效
- 至少 1 个真实 campaign 跑通 生成 -> 发布 -> 回填 -> 对比

## Phase 2：品牌知识库与 Guardrails（~4 周）

### 2.1 数据模型

在 [b2b_platform/schemas.py](apps/b2b_platform/schemas.py) 增强/新建：

- `**BrandGuardrail`**：`forbidden_expressions`、`must_mention_points`、`style_boundaries`、`risk_words`、`platform_guidelines`、`campaign_objectives`
- `**BrandVoice`**：`tone_spectrum`、`vocabulary_whitelist`、`vocabulary_blacklist`、`example_sentences`
- `**BrandProductLine`**（从 `BrandProfile.product_lines: list[str]` 升级为结构化对象）
- `**AudienceProfile`**：`personas`、`pain_points`、`decision_factors`、`taboo_topics`
- `**BrandObjective`**：`objective_type`、`target_metrics`、`constraints`

### 2.2 四阶段主链接入

在 [opportunity_to_plan_flow.py](apps/content_planning/services/opportunity_to_plan_flow.py) 的 `build_brief` / `build_strategy` / `build_plan` / `assemble_asset_bundle`：

- 从 `platform_store` 加载当前 `brand_id` 对应的 guardrails
- 传入 Council / Agent 的 `RequestContextBundle` 中（类似现有 `council_brief_snapshot`）
- `_apply_guardrails()` 方法在 apply/persist 前检查字段合规

### 2.3 Proposal/Apply 品牌阻断

在 [discussion.py](apps/content_planning/services/discussion.py)（或 proposal 相关逻辑）：

- `StageProposal` 增加 `guardrail_warnings: list[str]`、`blocked_by_guardrail: bool`、`brand_fit_score: float`
- Apply 时若 `blocked_by_guardrail=True` 返回 `422` + 阻断原因

### 2.4 Scorecard 增强

在 [evaluation/](apps/content_planning/evaluation/)：

- `DimensionScore` 增加 `brand_fit`、`brand_guardrail_fit`、`campaign_fit` 三个评分维度
- Rubric 版本升级

### 2.5 前端落点

- **新增 `brand_config.html` 模板**：品牌信息、产品线、guardrails、campaign 目标的配置页
- **四个对象工作台统一增强**（[content_brief.html](apps/intel_hub/api/templates/content_brief.html)、[content_strategy.html](apps/intel_hub/api/templates/content_strategy.html)、[content_plan.html](apps/intel_hub/api/templates/content_plan.html)、[content_assets.html](apps/intel_hub/api/templates/content_assets.html)）：
  - 右栏显示当前品牌上下文摘要
  - Guardrail warnings 提示条
  - Proposal diff 中标注 blocked fields

### 2.6 Phase 2 门槛验收

- 同一机会在不同品牌下生成显著不同的结果
- proposal/apply 能被品牌约束正确阻断
- 品牌负责人能直接看到"为什么这条建议不符合品牌"

## Phase 3：团队工作区 + 审批交付流（~4 周）

### 3.1 数据模型

在 [b2b_platform/schemas.py](apps/b2b_platform/schemas.py) 新增：

- `**ObjectAssignment`**：`assignment_id`、`workspace_id`、`object_type`、`object_id`、`assignee_user_id`、`assigned_by`、`role_hint`、`status`、`created_at`
- `**ObjectComment`**：`comment_id`、`workspace_id`、`object_type`、`object_id`、`author_user_id`、`content`、`created_at`
- `**WorkspaceTimelineEvent`**：`event_id`、`workspace_id`、`event_type`、`object_type`、`object_id`、`actor_user_id`、`detail`、`created_at`
- `**ApprovalRequest`**：`request_id`、`workspace_id`、`object_type`、`object_id`、`object_version`、`requested_by`、`requested_at`、`status`（pending/approved/rejected/withdrawn）、`reviewer_id`、`decision_at`、`notes`

### 3.2 统一对象状态机

所有策划对象（Brief/Strategy/Plan/AssetBundle）统一为：

```
new -> reviewed -> promoted -> in_planning -> ready -> approved -> exported -> published
```

- 在各 schema 中新增 `lifecycle_status` 字段（保留现有 `brief_status` / `strategy_status` 等向后兼容）
- 状态流转由 `OpportunityToPlanFlow` 统一管理

### 3.3 资产交付硬门槛

在 [opportunity_to_plan_flow.py](apps/content_planning/services/opportunity_to_plan_flow.py) / 路由层：

- `**ReadinessChecklist**` 模型：`export_readiness`（brand_fit >= 阈值、guardrail 无 block、scorecard >= 阈值）、`publish_readiness`、`approval_gate`
- `GET .../asset-bundle/{id}/export` 前检查 readiness；不通过返回 `409`
- `AssetBundle` 增强：变体管理（`variants: list[Variant]`）、导出记录（`export_records`）、发布记录、回流记录

### 3.4 API 路由

在 [routes.py](apps/content_planning/api/routes.py) / [app.py](apps/intel_hub/api/app.py)：

- `POST /objects/{type}/{id}/assign`
- `POST /objects/{type}/{id}/comments` / `GET .../comments`
- `POST /approvals/{id}/decision`（正式审批决策，区别于现有 flow 内审批）
- `GET /b2b/workspaces/{id}/timeline`（workspace 级事件流）

### 3.5 前端落点

- **新增 `workspace_home.html`**：Workspace Home，汇总卡片（机会数/进行中/待审/已发布）
- **新增 `opportunity_pipeline.html`**：Opportunity Pipeline，按 lifecycle_status 分列看板
- **新增 `review_approval.html`**：Review & Approval 队列页
- `**content_feedback.html` 增强**（Phase 1 已建）：合并为 Feedback & Knowledge 页
- **现有对象工作台补充**：assignment 标签、comment 面板、approval status 徽章、readiness gate 进度、timeline link

### 3.6 Phase 3 门槛验收

- 一个品牌团队能在系统内完成：机会发现 -> Brief/Strategy/Plan 编辑 -> 审批 -> 资产导出 -> 发布结果回流
- 不依赖人工改数据库或手工同步对象状态

---

## 总体验收标准

- 至少 3 个品牌试点可独立跑通完整链路
- 能量化展示：返工轮次下降、人工修改次数下降、ready 时间缩短、发布表现更稳定或更优

## 实施原则

- 每阶段过验收门槛后再进入下一阶段
- 复用现有主链和骨架，不做全量 Next.js 重写、不做 deer-flow/hermes 全量 runtime 接管、不做 Postgres 全量迁移
- 先补产品闭环，再补大底座
