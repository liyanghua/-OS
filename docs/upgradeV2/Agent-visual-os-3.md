 **Agent-native 内容经营 OS**。
 这个定位不是再做一个内容工具，而是把“热点、选题、达人、内容、投放、复盘”编译成一个可持续进化的经营系统。这个方向与当前平台现实也吻合：小红书官方已经把开放平台、聚光投放、蒲公英选号/标签图谱等能力平台化；同时，Agent 基础设施已经进入“多 Agent + 工具调用 + 评测”的成熟阶段。

# 0. 产品升级的核心判断

你当前的平台，本质上还是：

**热点抓取 → 爆款分析 → 选题生成 → 内容策划**

这已经有价值，但它仍偏“内容工厂前端”。
升级后的目标应该是：

**经营目标 → 机会发现 → 内容编译 → 达人/账号匹配 → 发布监控 → 投流建议 → 复盘沉淀 → 下一轮策略**

也就是从“内容策划平台”升级为：

## **Agent-native 内容经营 OS**

它的核心不是多一个 AI 生成功能，而是形成 4 层闭环：

1. **Signal Layer**：热点、竞品、平台信号、评论舆情
2. **Compilation Layer**：选题、爆款模式、品牌卖点、达人适配、创意 Brief
3. **Operation Layer**：达人协作、内容生产、发布监控、投流建议
4. **Learning Layer**：复盘归因、模式沉淀、策略更新、评测优化

# 1. 升级目标

## 1.1 产品定位

一句话：

**为品牌提供“人 + Agent”协同的内容经营操作系统，而不是单点内容生成工具。**

## 1.2 服务对象

第一阶段优先服务：

* 品牌操盘手
* 内容负责人
* 达人投放负责人
* 小红书内容运营负责人

第二阶段扩展：

* 代运营团队
* 创意团队
* KOC 运营团队
* 广告投放团队

## 1.3 解决的问题

从业务视角，系统升级后要解决 6 个问题：

1. 热点很多，但**不知道哪些与品牌真正相关**
2. 爆款很多，但**不知道哪些模式可以迁移**
3. 达人很多，但**不知道哪些达人最匹配当前内容方向**
4. 笔记很多，但**不知道哪些值得加热和放大**
5. 复盘很多，但**经验沉淀不下来**
6. AI 工具很多，但**无法串成经营闭环**

# 2. 升级路线图

## Phase 1：从“热点策划”升级到“机会编译”

目标：从只会产出选题，升级为产出**机会卡 + 选题卡 + 爆款模式卡**。

### 新增能力

* 热点抓取升级为多源 Signal Radar
* 爆款拆解升级为 Viral Pattern Extractor
* 品牌卖点与用户痛点融合
* 输出结构化 `OpportunityCard`

### 核心产物

* 热点机会卡
* 赛道机会卡
* 竞品内容机会卡
* 评论痛点机会卡

### 验收标准

* 机会卡可解释
* 有证据引用
* 有品牌相关性评分
* 有动作建议，不只是摘要

---

## Phase 2：从“机会编译”升级到“达人与内容协同”

目标：把内容机会与达人筛选、内容适配、建联推进串起来。

### 新增能力

* 样本账号喂养
* 相似账号召回
* 达人画像和品牌匹配打分
* 达人内容风格映射
* 建联话术生成
* 达人任务看板

### 核心产物

* CreatorFitCard
* OutreachTask
* ContentMatchPack

### 验收标准

* 选题可映射到达人池
* 达人可映射到内容建议
* 可形成建联与跟进清单

---

## Phase 3：从“内容协同”升级到“投后经营”

目标：接入发布后数据、投流建议和复盘回流，形成经营闭环。

### 新增能力

* 2h/6h/24h 指标监控
* 评论情绪与热点词分析
* 潜力笔记预测
* 投流建议
* 爆款归因
* 复盘资产沉淀

### 核心产物

* PostHealthCard
* BoostRecommendation
* ReviewCard
* PatternUpdate

### 验收标准

* 能区分“看起来热”与“值得加热”
* 能归因内容成功/失败
* 能反哺下一轮选题和选号

---

## Phase 4：从“经营闭环”升级到“多 Agent 操盘”

目标：从页面型工具升级为 Agent 编排系统。

### 新增能力

* 多 Agent 工作流
* 编排与审批
* 人工关键节点确认
* 可审计动作日志
* 评测与策略版本管理
* Skill 化输出

### 核心产物

* Plan Board
* Agent Run Log
* SkillSpec
* Eval Pack

### 验收标准

* 不同 Agent 分工清晰
* 输出稳定、可追溯
* 可在新品牌/新赛道快速复制

# 3. 升级后的 PRD

## 3.1 产品愿景

打造一个以品牌目标为起点、以内容经营资产为中枢、以多 Agent 协同为手段、以复盘进化为终点的内容经营 OS。

## 3.2 核心原则

### 1）AI-native，但不是 chat-first

入口可以有聊天，但主交互应是：

* 机会卡
* 选题卡
* 达人卡
* 监控卡
* 复盘卡
* Plan Board

### 2）资产优先

所有输出都要沉淀为资产，而不是停留在对话里。

### 3）证据优先

每张卡必须带证据，不允许“空洞建议”。

### 4）人工拍板关键节点

* 是否建联
* 是否发布
* 是否投流
* 是否大预算放大
* 是否采纳策略更新

### 5）平台共存

优先走平台开放与受控能力，不把高风险自动化作为主路径。小红书官方现有公开体系本身就是“开放平台 + 聚光 + 蒲公英 + 服务市场”的平台化结构。([小红书][1])

---

## 3.3 核心用户故事

### 用户故事 A：内容负责人

“我想知道这周小红书哪些热点与我品牌相关，哪些可做，为什么可做，应该找谁做。”

系统输出：

* 热点榜
* 品牌相关机会卡
* 推荐达人类型
* 推荐选题方向

### 用户故事 B：达人投放负责人

“我想用历史合作账号和竞品账号找出一批更适合当前 campaign 的达人，并自动生成建联建议。”

系统输出：

* 相似账号池
* 达人优先级排序
* 风格匹配解释
* 建联任务清单

### 用户故事 C：操盘手

“我想知道过去 24 小时哪些笔记值得加热，为什么，建议怎么投。”

系统输出：

* 潜力笔记排行榜
* 投流建议卡
* 风险说明
* 预计测试预算

### 用户故事 D：老板/总监

“我只想看到机会、风险、放大点，以及下一轮应该拍板什么。”

系统输出：

* 本周经营机会摘要
* 风险卡
* Top 爆款模式
* 待拍板事项

# 6. Agent 架构

公开的 Agent 方向已经很明确：Agent 应该是可调用工具、可协作、可保留状态、可评测的多步系统，而不是单纯的 prompt 拼接。OpenAI 官方 Agents SDK、Responses API、Agent Builder 都在围绕这类能力演进。([OpenAI开发者][2])

## 6.1 建议的 Agent 拆分

### 1）Signal Hunter

负责：

* 抓热点
* 抓竞品
* 抓评论信号
* 归并成 SignalEvent

### 2）Opportunity Compiler

负责：

* 把 SignalEvent 编译成 OpportunityCard
* 做品牌相关性匹配
* 输出动作建议

### 3）Creator Scout

负责：

* 样本达人画像提取
* 相似达人召回
* 达人匹配打分

### 4）Outreach Writer

负责：

* 生成建联话术
* 跟进话术
* 报价沟通模板

### 5）Viral Pattern Miner

负责：

* 爆款模式抽取
* Hook/结构/视觉模板化

### 6）Topic Planner

负责：

* 机会转选题
* 选题转内容 Brief

### 7）Post Monitor

负责：

* 指标更新
* 舆情分析
* 潜力笔记识别

### 8）Boost Advisor

负责：

* 投流建议
* 首轮预算建议
* 放大/止损建议

### 9）Review Synthesizer

负责：

* 归因分析
* 资产更新
* 下轮建议

### 10）Plan Orchestrator

负责：

* 串联全链路
* 控制人工审批节点
* 管理版本与日志

# 7. 技术方案

## 7.1 总体技术架构

```text
外部数据源 / 官方平台 / 第三方导入 / 浏览器采集
                ↓
         Ingestion Layer
                ↓
      Normalization + Entity Layer
                ↓
   Signal Store + Content Asset Store + Creator Store
                ↓
        Agent Runtime / Workflow Engine
                ↓
 Opportunity / Creator / Topic / Monitor / Review APIs
                ↓
         Web Workbench (Vue3 + TS)
```

---

## 7.2 数据接入层

### 数据源分层

#### A. 官方/授权层

* 小红书开放平台
* 聚光投放平台
* 蒲公英合作平台
* 服务市场能力

#### B. 第三方导入层

* 新红 / 灰豚 / CSV / Excel
* 竞品监测表
* 历史合作达人表

#### C. 浏览器辅助采集层

* 仅抓公开页面或授权页面
* 不把高风险批量无人值守操作作为主路径

公开文档显示，小红书开放平台提供开发者文档、授权流程、签名算法、SDK 等；聚光提供一站式营销投放；蒲公英则已有智能选K、博主标签图谱等能力。([小红书][1])

---

## 7.3 存储层

建议四类存储：

### 1）Signal Store

存热点、竞品、评论舆情、趋势事件

### 2）Asset Store

存选题库、爆款模式库、复盘库、品牌知识

### 3）Creator Store

存达人基础信息、匹配分、合作历史

### 4）Metric Store

存 post snapshot、评论聚类、投后表现

---

## 7.4 检索与编译层

### 检索

* keyword search
* embedding search
* graph-style relation search

### 编译

* Signal → Opportunity
* Opportunity → Topic
* Topic → ContentBrief
* Post → ReviewCard
* ReviewCard → Asset Update

这里建议所有编译结果都使用结构化输出，不要直接放纯文本长答案。

---

## 7.5 Agent Runtime

建议路线：

* 第一阶段：后端自建 workflow orchestration
* 第二阶段：接入更规范的 Agent runtime
* 第三阶段：Skill 化与评测化

OpenAI 官方当前已经明确建议新项目采用 Responses API，作为更 agentic 的基础接口，并且 Agents SDK 支持多步工具调用；Agent Builder 则适合先设计、调试、导出工作流。([OpenAI开发者][3])

# 8. 关键流程设计

## 8.1 热点机会编译流程

```text
抓取热点/爆款/评论信号
→ 信号去重/归一
→ 品牌相关性匹配
→ 用户痛点与卖点映射
→ 输出 OpportunityCard
→ 人工确认是否进入 Topic Pipeline
```

## 8.2 达人匹配流程

```text
输入样本账号
→ 提取风格与人群画像
→ 相似达人召回
→ 打分与解释
→ 进入 CreatorFitCard
→ 人工确认是否发起建联
```

## 8.3 内容编译流程

```text
OpportunityCard
→ ViralPattern 匹配
→ TopicCard 生成
→ ContentBrief 生成
→ 指派达人/KOC
→ 发布跟踪
```

## 8.4 投后闭环流程

```text
发布后 2h/6h/24h 指标监控
→ 舆情与评论分析
→ 潜力笔记预测
→ 投流建议
→ 复盘归因
→ 更新资产库
```

# 9. 评测体系

要升级成 OS，必须有 eval，不然只是“智能页面”。

## 9.1 机会卡评测

* 品牌相关性
* 可执行性
* 证据充分性
* 新颖性
* 优先级合理性

## 9.2 达人匹配评测

* 匹配准确率
* 人工接受率
* 后续建联转化率
* 合作 ROI

## 9.3 选题评测

* 选题采纳率
* 选题发布率
* 发布后互动率
* 收藏率/评论率

## 9.4 潜力笔记评测

* 24h 命中率
* 值得加热命中率
* 误判率
* 推荐预算有效率

## 9.5 复盘评测

* 归因解释满意度
* 下轮建议命中率
* 资产复用率

# 12. 最后一句定义

把你现在的平台升级成 Agent-native 内容经营 OS，核心不是“功能加法”，而是完成三个跃迁：

1. **从热点工具 → 机会编译系统**
2. **从内容策划工具 → 达人-内容-投放协同系统**
3. **从单轮 AI 生成 → 可沉淀、可评测、可进化的经营 OS**

这条路线既贴合你现在已有积累，也符合平台现实：平台在做官方能力中台化，品牌需要自己的经营编译层，Agent 技术则已经足够支撑多步、可调用、可评测的系统化产品。([小红书][1])
